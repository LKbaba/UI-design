# 需求澄清问题清单（Discover 阶段）

> 什么时候读这份：进入 discover 阶段前。它是 SKILL.md 里 P1 需求澄清的详细执行手册。

---

## 上下文优先级

获取设计上下文时，按以下优先级：

1. **用户的设计系统** — 已有 token / 组件库 / Figma / 品牌手册 → 直接用
2. **代码库** — `package.json`、`tailwind.config.*`、全局样式 → 读取实际依赖
3. **已发布产品** — 用户线上产品截图 → 提取颜色/字体/间距
4. **品牌资料** — logo、品牌色、字体 → 用 `brand-tokens-template.md` 落到文档
5. **竞品参考** — 用户指定的对标产品 → 仅学习布局策略，不抄视觉
6. **兜底** — 无任何参考 → 用中性设计系统（灰色系 + 系统字体），等用户补充

**硬规则**：
- ❌ 禁止凭记忆猜品牌颜色（用 WebSearch 或读代码确认）
- ❌ 禁止用 CSS 形状代替真实产品图
- ❌ 禁止混淆演示品牌和用户品牌

## 路由成型问题（必问，每次 1 个）

这些问题决定后续加载哪些 references、用什么技术栈、走什么验证路径。用 `AskUserQuestion`（或平台等效工具）逐条问，每次给 2-4 个选项 + freeform。

### Q1 · 输出类型

> 你需要我做什么？

| 选项 | 说明 |
|------|------|
| 完整页面 | 落地页、产品首页、仪表盘等 |
| Section / 区块 | Hero、Features、Pricing 等独立区块 |
| 组件 | Button、Card、Modal 等可复用组件 |
| 交互原型 | 带状态/流程的可交互原型 |
| 设计评审 | 审查现有界面，给出改进建议 |
| 动效 / 微交互 | 转场、hover 效果、加载动画等 |

### Q2 · 技术栈

> 你的项目用什么技术栈？（我会读你的 package.json 确认）

| 选项 | 说明 |
|------|------|
| HTML 静态原型 | 纯 HTML + CSS（+ 可选 JS），无构建工具 |
| React / Next.js | React 生态（含 Remix、Gatsby 等） |
| Vue / Nuxt | Vue 生态 |
| 用户现有项目 | 我来读你的代码确认 |
| 其他 | 请告诉我 |

### Q3 · 任务状态

> 这个任务的起点是什么？

| 选项 | 说明 |
|------|------|
| 全新任务 | 从零开始 |
| 局部编辑 | 改现有页面的某个部分 |
| 已批准迭代后续 | 上次做的已确认，继续下一步 |
| 重做 / 推翻 | 上次方向不对，重新来 |

### Q4 · 可用上下文

> 你有哪些可以给我的参考资料？

| 选项 | 说明 |
|------|------|
| 设计系统 / Figma | 已有完整设计系统 |
| 代码库 | 现有项目代码 |
| 截图 / 线上产品 | 可以看到的现有产品 |
| 品牌资料 | logo / 品牌手册 / 色板 |
| 对标产品 | 想参考某个产品的风格 |
| 无参考 | 自由发挥 |

### Q5 · 主要设计风险

> 你最担心这个设计的哪个方面？

| 选项 | 说明 |
|------|------|
| 布局 | 信息怎么排、空间怎么分 |
| 排版 | 字体、字号、行高、层级 |
| 颜色 | 配色方案、品牌一致性 |
| 信息层级 | 主次关系、用户注意力引导 |
| 交互 | 状态切换、微交互、动效 |
| 品牌调性 | 整体感觉是否匹配品牌 |

## 深度问题（可选，视情况追问）

路由成型问题问完后，如果信息还不够，可以追问以下：

### 受众

- 谁是主要用户？（企业决策者 / C 端消费者 / 内部员工 / 开发者）
- 用户的技术水平？（技术小白 / 普通用户 / 技术人员）
- 使用场景？（办公桌前 / 通勤路上 / 会议室演示）

### 范围

- 需要响应式吗？（仅桌面 / 仅移动 / 全响应式）
- 有多少个页面/视图？
- 需要暗色模式吗？

### 硬约束

- 必须复用某组件库？（shadcn/ui / Ant Design / MUI / Element Plus / 自建）
- 有性能约束吗？（首屏 ≤ 3s / 无外部依赖 / 必须 SSR）
- 有无障碍要求吗？（WCAG AA / AAA / 无特殊要求）

### 信息架构

- 页面的核心行动号召（CTA）是什么？
- 用户从进入到完成目标的路径？
- 有多少个 section？每个的优先级？

### 品牌 Tokens

- 主色 / 辅色 / 中性色？
- 字体族？（如有品牌字体）
- 间距偏好？（紧凑 / 标准 / 宽松）
- 圆角偏好？（直角 / 小圆角 / 大圆角 / 全圆）

如果用户提到品牌色 / 字体 / token，加载 `references/brand-tokens-template.md` 让用户填空。

## 品牌资产获取协议

当任务涉及真实品牌时，执行以下流程：

1. **资产识别** — 列出需要的品牌素材（logo、产品图、品牌色、字体）
2. **实际获取** — 用 WebSearch 或让用户提供，不凭记忆
3. **验证** — 对比官网/官方资料确认颜色和字体准确
4. **提取颜色** — 从实际素材取色，记录 HEX 值
5. **固化** — 写入 `DESIGN.md` 的品牌 tokens 章节

**失败案例提醒**：
- 凭记忆猜品牌色 → 大概率猜错（如某品牌主色记成橙色，实际是 `#1783FF`）
- 用演示品牌色替代真实品牌色 → 交付物和用户预期完全不符
- 用 CSS 画产品图 → 不如干净的灰色占位块
